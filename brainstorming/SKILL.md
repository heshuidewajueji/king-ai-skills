# Brainstorming

把模糊想法捋成清晰方案，适用于需求发散、PRD 框架、多方案对比、问题拆解等场景。

## 核心流程

### 1. 发散阶段 — 不评判，先堆量

**策略**：先用「有多少种方式」提问，强制发散。

```
用户：TBOX 要加远程诊断功能

→ 追问：这个功能可以有哪些实现路径？
→ 追问：从技术维度分？有无监管要求？从用户场景分？
→ 追问：竞品怎么做的？
```

**原则**：发散阶段不否定任何想法，全部记录。

### 2. 收敛阶段 — 分维度评估

**策略**：用「Kano模型」或「重要性 vs 实施难度」矩阵收敛。

```
收敛维度：
- 用户价值（高/中/低）
- 技术难度（简单/中等/复杂）
- 商业价值（高/中/低）
- 监管/合规（强相关/弱相关/无关）
```

**原则**：给出 2-3 个推荐方案，说明推荐理由。

### 3. 输出结构化方案

```
## 方案对比

| 维度 | 方案A | 方案B | 方案C |
|------|-------|-------|-------|
| 核心思路 | | | |
| 用户价值 | | | |
| 技术成本 | | | |
| 推荐度 | ⭐⭐⭐ | ⭐⭐ | ⭐ |

## 推荐方案
（推荐理由 + 实施路径 + 潜在风险）
```

## PM 场景模板

### 需求模糊期

```
输入：「用户希望能 XXX」

输出结构：
1. 澄清问题：XXX 的具体场景是？
2. 用户是谁：他们的核心诉求是？
3. 解决方案发散：至少 3 种路径
4. 收敛建议：推荐路径 + 理由
```

### PRD 框架搭建

```
输入：一个功能点

输出结构：
1. 背景/问题
2. 用户故事
3. 功能范围（要做/不做）
4. 流程图/原型描述
5. 验收标准
6. 风险点
```

### 竞品分析发散

```
输入：竞品功能截图/描述

输出结构：
1. 功能拆解：核心模块有哪些
2. 优劣势分析
3. 我方机会点
4. 可借鉴设计
```

## 使用提示

- 遇到模糊需求，先追问「具体场景是什么」再输出
- 多方案对比时，给出明确推荐，避免用户选择困难
- 定期把有效 brainstorm 结果写回 vault，形成知识积累
