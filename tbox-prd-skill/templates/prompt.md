# TBOX PRD 生成 Prompt 模板

> 本文件是 AI Agent 生成 TBOX PRD 时的标准 Prompt 模板。
> 使用时：填入【】中的变量 → 按顺序执行各步骤。

---

## 场景一：首次生成完整 FRS（客户提供完整信息）

```
你是汽车电子 TBOX 领域专家，负责为客户生成功能需求规范文档（FRS）。

【输入信息】
- 客户：{客户名称}
- 车型：{车型代号}
- 目标市场：{国内新能源 / 出口（欧盟/印度/其他）/ 商用车}
- 通信制式：{4G Cat.1 / 4G Cat.4 / 5G}
- CAN 速率：{500K / 250K / CAN FD}
- 接入 ECU：{VCU / BMS / MCU / BCM}
- 定制功能：{远程解锁 / 空调 / 数字钥匙 / OTA / 无}
- 必选法规：{GB/T 32960.3-2025 / TAF 安全要求}

【第一步】读取领域知识
读取 ~/.openclaw/workspace-skills/tbox-prd-skill/knowledge/domain.md
重点关注：系统架构（1.1节）、功能模块体系（2节）、CAN接口（3节）

【第二步】读取标准规范
读取 ~/.openclaw/workspace-skills/tbox-prd-skill/knowledge/standards.md
重点关注：GB/T 32960 合规检查要点（1.2节）、信息安全章节（2节）

【第三步】按模板生成
读取 ~/.openclaw/workspace-skills/tbox-prd-skill/templates/frs.md
按模板格式生成完整 FRS，【】内变量用输入信息替换

【第四步】补充法规章节
- 国内新能源：必须补充 GB/T 32960.3 合规声明章节
- 欧盟出口：必须补充 UNECE R155/R156 信息安全章节
- 有定制 OTA：必须补充 FR-07 远程升级章节

【第五步】输出
生成完整 FRS Markdown 文档，直接输出全文。
```

## 场景二：评审现有 PRD

```
你是汽车电子 TBOX 领域专家，负责评审现有功能需求规范文档。

【待评审文档】
粘贴现有 PRD 全文内容

【评审维度】
1. GB/T 32960.3-2025 合规性
   - [ ] 是否覆盖所有必选功能（数据采集/上报/登入登出/报警/存储）
   - [ ] 通信协议是否对齐 GB/T 32960.3 报文格式
   - [ ] 数据加密是否明确 SM2/SM3/SM4

2. TAF 安全技术要求合规性
   - [ ] 是否有独立的信息安全章节
   - [ ] CAN 总线隔离是否描述
   - [ ] 签名认证机制是否定义

3. 需求描述规范性
   - [ ] 每条需求是否包含：触发条件 + 行为 + 验收条件
   - [ ] 是否使用 Gherkin 格式（Given-When-Then）
   - [ ] 定量指标是否明确（时间/频率/数值）

4. 接口完整性
   - [ ] CAN 接口是否列出所有信号
   - [ ] 信号是否包含：CAN ID / 周期 / 物理值换算
   - [ ] OTA 接口是否定义升级流程和校验机制

【输出格式】
评审结论：
- 合规性：{通过/有条件通过/不通过}
- 主要问题：{列表}
- 修改建议：{每条问题的具体修改方向}
```

## 场景三：根据 CAN 矩阵生成信号映射表

```
你是汽车电子 TBOX 领域专家，负责将客户的 CAN 矩阵转换为 TBOX 接口定义。

【输入】
粘贴客户的 CAN 矩阵（Excel 或文本格式）

【处理步骤】
1. 解析每条 CAN 帧：ID / DLC / 信号列表
2. 识别 GB/T 32960 标准帧（0x01~0x08）
3. 识别客户定制帧
4. 按以下格式输出信号映射表：

【输出：CAN 信号映射表】

| 序号 | CAN ID | 信号名称 | 周期(ms) | 物理范围 | 换算公式 | 来源ECU | GB/T 32960对应 | 备注 |
|------|--------|---------|---------|---------|---------|---------|-------------|------|
| 1 | 0x01 | VCU_VehicleSpeed_kmh | 1000 | 0~220km/h | Raw/50 | VCU | 必选 | |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

【验证】
- 标准帧是否覆盖完整
- 定制帧是否标注客户来源
- 所有信号是否包含换算公式
```

## 场景四：生成 SRS（软件需求规范）

```
你是汽车电子 TBOX 软件架构专家，负责从 FRS 生成软件需求规范（SRS）。

【输入】
粘贴已完成评审的 FRS 文档

【处理步骤】
1. 将 FRS 每条功能需求映射到软件需求
2. 按 AUTOSAR 分层：Application / RTE / BSW / MCAL
3. 识别软件组件（SWC）及其接口
4. 确定 ASIL 等级分配
5. 按 GB/T 34590.6 要求输出 SRS

【SRS 必须包含的章节】
1. 软件概述（范围、约束、依赖）
2. 软件需求（按功能模块分解）
   - FR → SR 追踪
   - 每个 SR 必须包含：ID / 描述 / 输入 / 输出 / 前置条件 / 后置条件 / 性能要求
3. 软件架构（AUTOSAR 分层）
   - RTE 接口定义
   - BSW 模块选型
   - 通信矩阵（CAN ID 分配）
4. 接口规格
   - CAN Interface
   - NVRAM Interface
   - COM Interface
5. 非功能需求（响应时间 / 存储占用 / 功耗）
6. 需求追踪矩阵（FR → SR → SWC）
```

```

- 已确认项：{双方已确认的内容}
- 待确认项：{需进一步确认的内容}

【处理步骤】
1. 基于 {客户名称} 的历史项目经验（如有）
2. 按 templates/frs.md 模板生成框架
3. 【待确认】项用红色标注
4. 附上「确认清单」附录

【输出】
生成 FRS 框架后，附上：

| 序号 | 确认项 | 选项 | 确认人 | 截止日期 |

| 1 | 通信制式 | ☐ 4G Cat.1  ☐ 4G Cat.4  ☐ 5G | | |
| 2 | CAN 波特率 | ☐ 500K  ☐ 250K  ☐ CAN FD | | |
| 3 | OTA 需求 | ☐ 需要  ☐ 不需要 | | |
| 4 | 定位精度 | ☐ 2.5m  ☐ 1m（RTK） | | |
```
