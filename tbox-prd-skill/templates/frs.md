## 3. 功能需求

### 3.1 FR-01 车辆数据采集

#### FR-01-001 CAN 总线数据采集
**需求描述**：
TBOX 应从整车 CAN 总线采集以下数据，周期 `{1s / 10s}`：

| CAN ID | 数据内容 | 周期 | 来源 ECU | 备注 |
|--------|---------|------|---------|------|
| 0x01 | {整车数据} | 1s | VCU | 必选 |
| 0x02 | {电池数据} | 1s | BMS | 必选 |
| 0x03 | {驱动数据} | 1s | MCU | 必选 |
| {xxx} | {自定义信号} | {x}s | {ECU} | {客户定制} |

**验收条件**：
- [ ] CAN 总线波特率 `{500K / 250K}` 正确配置
- [ ] 所有信号物理值换算正确（附换算公式）
- [ ] 信号超时（≥`{3}` 个周期无更新）触发故障记录

#### FR-01-002 充电状态检测
**需求描述**：
当检测到充电枪连接信号时，TBOX 应自动切换到充电模式：

| 条件 | 行为 |
|------|------|
| 慢充枪连接 | 上报 `ChargingType=2`，周期切换到 `{10s}` |
| 快充枪连接 | 上报 `ChargingType=1`，周期切换到 `{1s}` |
| 充电结束 | 3min 内无充电电流 → 恢复默认周期 |

---

### 3.2 FR-02 数据通信

#### FR-02-001 云平台登入登出
**需求描述**：
TBOX 上电后应自动登入监控平台，下电前应主动登出：

| 事件 | 触发条件 | 时限要求 |
|------|---------|---------|
| 登入 | KL.15 ON 或定时 | 登入后 `{30s}` 内发送 |
| 重登入 | 网络中断恢复 | `{60s}` 内自动重连 |
| 登出 | KL.15 OFF | 延迟 `{10s}` 后发送 |
| 心跳 | 登入后持续 | `{30s}` 间隔 |

**GB/T 32960.3 合规**：
- 登入命令：`0x01`
- 实时数据：`0x02`
- 报警数据：`0x04`
- 登出命令：`0x03`
- 校验码：CRC-8

#### FR-02-002 数据加密
**需求描述**：
所有与云平台的数据交互必须使用国密算法加密：

| 算法 | 用途 | 密钥长度 |
|------|------|---------|
| SM2 | 双向身份认证与会话密钥交换 | 256bit |
| SM4 | 数据传输加密 | 128bit |
| SM3 | 消息完整性校验（MAC）| — |

**会话密钥更新**：每 `{7天}` 或 `{10000次}` 会话更新一次。

### 3.3 FR-03 定位功能

#### FR-03-001 卫星定位
**需求描述**：
TBOX 应支持 `{GPS+BDS / GPS+BDS+GLONASS+Galileo}` 多系统定位：

| 指标 | 要求 |
|------|------|
| 冷启动定位时间 | ≤ `{60s}`（CEP < 5m）|
| 热启动定位时间 | ≤ `{5s}` |
| 定位模式 | {单点定位 / RTK（可选）} |
| 定位数据周期 | ≤ `{1s}` |
| 高度精度 | ≤ `{10m}` CEP |
| 速度精度 | ≤ `{0.2m/s}` |

#### FR-03-002 GNSS 异常处理
**需求描述**：
当 GNSS 信号丢失时，TBOX 应通过惯性导航（INS）航位推算：

| 场景 | 行为 | 时限 |
|------|------|------|
| GNSS 失锁 < 60s | 航位推算，标记 `PosSource=INS` | 持续 |
| GNSS 失锁 ≥ 60s | 停止航位推算，标记 `PosValid=0` | 触发 |
| GNSS 恢复 | 自动切回卫星定位，清除 INS 标记 | < 5s |

---

### 3.4 FR-04 本地数据存储

#### FR-04-001 断网数据缓存
**需求描述**：
当与云平台网络中断时，TBOX 应本地缓存未成功上报的数据：

| 指标 | 要求 |
|------|------|
| 缓存容量 | ≥ `{72小时}` 数据（按 1 条/秒计）|
| 存储介质 | NAND Flash 或 eMMC（带 ECC）|
| 循环策略 | 满则覆写最旧数据 |
| 补传机制 | 网络恢复后自动补传，按时间顺序 |

**存储数据项**：
```
时间戳 | CAN数据(原始) | 定位数据 | 报警状态 | 通信状态
```

#### FR-04-002 休眠数据保留
**需求描述**：
TBOX 进入休眠前，应将当前状态写入非易失存储：

| 数据项 | 内容 |
|--------|------|
| 上次定位 | 经纬度/时间/来源 |
| 未上报记录数 | 整数 |
| SN / ICCID | 设备身份信息 |

---

### 3.5 FR-05 报警功能

#### FR-05-001 报警触发与上报
**需求描述**：
当检测到以下条件时，TBOX 应在 `{30s}` 内向云平台发送报警数据（0x04）：

| 报警类型 | 触发条件 | 优先级 |
|---------|---------|--------|
| 温度报警 | 电池温度 > `{50℃}` 或 < `{-20℃}` | P1 |
| 电压报警 | 单体电压 > `{3.65V}` 或 < `{2.8V}` | P1 |
| SOC 报警 | SOC < `{15%}` 或 SOC < `{5%}` | P2 |
| CAN 通信故障 | ≥ `{3}` 个周期无有效 CAN 数据 | P2 |
| GNSS 故障 | 持续 `{60s}` 无法定位 | P3 |
| TBOX 防拆报警 | 拆除触发器激活 | P1 |

**报警数据格式**（按 GB/T 32960.3）：
```
报警等级(1字节) | 报警类型(1字节) | 报警时间 | 报警时的位置 | 附加数据
```

### 3.6 FR-06 远程控制 `{[客户定制]}`

#### FR-06-001 远程解锁 `{[如需要]}`
**需求描述**：
收到云平台远程解锁命令后，TBOX 应：

1. 验证命令签名（SM2）
2. 通过 CAN 总线发送解锁指令到 BCM
3. 等待 BCM 确认（超时 `{3s}`）
4. 上报执行结果到云平台

| 结果 | 响应 |
|------|------|
| 成功 | 发送 `UnlockResult=0`，记录解锁时间 |
| BCM 无响应 | 发送 `UnlockResult=1`，`{3s}` 后重试 `{2}` 次 |
| 签名验证失败 | 丢弃命令，记录安全日志 |

#### FR-06-002 远程空调 `{[如需要]}`
**需求描述**：
收到云平台空调控制命令后，TBOX 应通过 CAN 总线转发到 HVAC 控制器：

| 命令 | CAN 信号 | 数值 |
|------|---------|------|
| 开启空调 | `HVAC_Cmd` | `0x01` |
| 关闭空调 | `HVAC_Cmd` | `0x00` |
| 设置温度 | `HVAC_TempSet` | `{16~30}℃` |
| 设置风向 | `HVAC_FanDir` | `{0:自动,1~4:固定风向}` |

### 3.7 FR-07 OTA 升级 `{[如需要]}`

#### FR-07-001 远程固件升级
**需求描述**：
TBOX 应支持远程 OTA 升级：

| 项目 | 要求 |
|------|------|
| 升级通道 | 4G 移动通信（首选）/ WiFi（备选）|
| 差分升级 | 支持（节省流量）|

| 包校验 | MD5 + 版本号双重校验 |
| 签名验证 | 必须基于 OEM 数字证书 |
| 升级时间 | 默认 `{23:00~05:00}` 免打扰时段 |
| ECU 联动 | {支持 / 不支持}（如支持，需定义升级顺序）|
| 失败处理 | 自动回滚到前一版本 |
| 结果上报 | 上报升级结果（含失败原因） |

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 要求 |
|------|------|
| 系统启动时间 | ≤ `{30s}`（从 KL.15 ON 到 CAN 收发就绪）|
| CAN 数据采集延迟 | ≤ `{100ms}`（从 CAN 帧到本地缓存）|
| 定位数据延迟 | ≤ `{200ms}` |
| 4G 发送延迟 | ≤ `{500ms}`（从数据就绪到 TCP 发送完成）|
| OTA 下载速度 | ≥ `{2Mbps}` |
| 并发 CAN 帧处理 | ≥ `{500帧/秒}` |

### 4.2 可靠性要求

| 指标 | 要求 |
|------|------|
| MTBF | ≥ `{30000小时}` |
| 使用寿命 | ≥ `{10年}` 或 `{15万公里}` |
| 设计寿命周期 | ≥ `{10000次}` 电源循环 |
| 固件存储寿命 | ≥ `{10年}`（非易失存储）|
| CAN 总线错误容忍 | ≥ `{99.99%}` 有效帧 |

### 4.3 安全性要求

| 指标 | 要求 |
|------|------|
| 身份鉴别 | 双向 SM2 证书认证 |
| 数据加密 | SM4，全流量加密 |
| 软件签名 | 全量固件 + APP 必须 OEM 签名 |
| 防回滚 | 不允许安装旧版本固件 |
| 防拆机制 | 壳体拆解 → 清除加密密钥 + 报警 |
| 安全启动 | MCU Bootloader 必须验签后执行 |
