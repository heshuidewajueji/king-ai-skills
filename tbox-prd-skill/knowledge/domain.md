## 1. TBOX 系统架构

### 1.1 硬件架构（MPU + MCU 双核）

```
┌─────────────────────────────────────────────────────────────┐
│                      TBOX 硬件架构                          │
├──────────────────────┬──────────────────────────────────────┤
│         MPU          │              MCU                    │
│   （应用处理器）       │           （通信处理器）             │
│  Android / Linux     │          实时操作系统               │
├──────────────────────┼──────────────────────────────────────┤
│  4G/5G 移动通信单元   │  CAN 总线控制器（支持 CAN FD）       │
│  GPS / BDS 定位单元   │  电源管理（宽压 9~36V）             │
│  蓝牙 / WiFi 模块     │  看门狗（硬件监控）                  │
│  以太网 PHY          │  加解密芯片（SE / TPM）              │
│  存储（eMMC / NOR） │  实时时钟（RTC，带超级电容）          │
│  音频解码（TTS）      │  状态指示 LED                       │
└──────────────────────┴──────────────────────────────────────┘
```

**MPU 职责**：应用程序、远程控车、GPS 解算、蓝牙协议、OTA 升级
**MCU 职责**：CAN 总线数据采集、电源管理、车辆休眠/唤醒、实时性保证

### 1.2 系统原理

```
手机 APP
    ↓
云平台（TSP）
    ↓  TCP/SSL
TBOX（4G模块）
    ↓ CAN总线
┌───┐ ┌───┐ ┌───┐ ┌───┐ ┌───┐
│VCU│ │BMS│ │MCU│ │BCM│ │EMS│  ← 车辆各ECU
└───┘ └───┘ └───┘ └───┘ └───┘
```

手机发送控制命令流程：
`手机 → 云平台 → TBOX → CAN总线 → 目标ECU → 执行机构 → 结果回传`

## 2. 功能模块体系

| 模块 | 功能 | 优先级 | 依据标准 |
|------|------|--------|---------|
| 车辆数据采集 | 车速/里程/SOC/温度/报警等 | 必须 | GB/T 32960.2 |

| 远程诊断 | OBD 读取故障码/实时数据 | 可选 | OEM定制 |

## 3. CAN 总线接口规范

### 3.1 标准接入信号（GB/T 32960 定义）

| CAN ID | 信号内容 | 周期 | 来源ECU |
|--------|---------|------|---------|
| 0x01 | 整车数据（车速、里程、运行状态）| 1s | VCU |
| 0x02 | 电池数据（SOC、单体电压、温度）| 1s | BMS |
| 0x03 | 驱动电机数据（转速、扭矩、温度）| 1s | MCU |
| 0x04 | 燃料电池数据（电压、电流、功率）| 1s | FCU |
| 0x05 | 发动机数据（转速、油门、温度）| 1s | EMS |
| 0x06 | 车辆位置数据 | 1s | GPS模块 |
| 0x07 | 极值数据（最高最低电压温度）| 1s | BMS |
| 0x08 | 报警数据 | 触发上报 | BMS/VCU |

### 3.2 CAN 信号命名规范

```
格式：{ECU}_{SignalName}_{物理单位（可选）}

示例：
  VCU_VehicleSpeed_kmh        车速（km/h）
  VCU_TotalOdometer_m         总里程（m）
  BMS_StateOfCharge_pct       SOC（%）
  BMS_BattMaxCellTemp_degC    最高温度（℃）
  MCU_DriverMotorTorque_Nm    驱动电机扭矩（Nm）
  VCU_ChargingStatus          充电状态（0:未充电 1:快充 2:慢充）
```

## 4. 通信协议

### 4.1 GB/T 32960.3 协议结构

```
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ 起始符  │ 命令单元│ 数据单元长度 │ 数据单元 │ 校验码  │ 结束符  │
│ 0x23   │ 1字节  │  2字节  │ N字节  │ 1字节  │ 0x0D   │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┴────────┘
         ↑                            ↑
      登录:0x01                   CRC-8校验
      实时:0x02
      登出:0x03
      ...
```

### 4.2 数据加密

- **必选**：GB/T 32960 要求使用国密 SM2/SM3/SM4
- **可选**：TLS 1.2+（欧洲出口车型，UNECE R155）

## 5. 电源管理与工作模式

### 5.1 工作模式

| 模式 | MPU状态 | CAN状态 | 通信 | 典型功耗 |
|------|---------|---------|------|---------|
| 正常工作 | 运行 | 收发 | 4G在线 | 200~500mA @12V |
| 休眠（KL.30）| 待机 | 仅监听唤醒帧 | 4G保活 | 3~10mA @12V |
| 深睡（运输模式）| 下电 | 关闭 | 关闭 | ≤2mA @12V |
| 诊断模式 | 运行 | 收发 | USB/以太网 | 由外部供电 |

### 5.2 唤醒条件

| 条件 | 来源 | 优先级 |
|------|------|--------|
| KL.15 点火唤醒 | 硬线（ACC/IGN）| 最高 |
| CAN 总线唤醒 | 特定 CAN ID | 中 |
| 远程唤醒（短信/APP）| 4G 模块 | 低 |
| 定时唤醒 | MCU RTC | 低 |
| 调试接口唤醒 | USB/串口 | 仅诊断模式 |

## 6. OTA 升级

### 6.1 升级流程

```
1. 云平台推送升级任务（含目标版本、包URL、MD5）
2. TBOX MCU 接收并存储升级指令
3. TBOX MPU 下载升级包（支持断点续传）
4. MD5 校验通过后，通知 MCU 执行重启升级
5. MCU 切换到 Bootloader，分区切换
6. 升级失败 → 自动回滚到旧分区
7. 上报升级结果到云平台
```

### 6.2 ECU 联动 OTA（进阶）

- 整车 OTA 需 TBOX 充当网关，同时升级 VCU/BMS/MCU
- 需定义升级顺序（通常 VCU 先升，TBOX 最后）
- 需支持差分升级（节省流量）

| MTBF | ≥30000h（车载级要求）| |
