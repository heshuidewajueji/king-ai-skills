## 1. GB/T 32960 系列（国内新能源强制）

> **法律地位**：工信部强标，未通过认证不得上市销售

| 标准号 | 名称 | 适用范围 | 关键章节 |
|--------|------|---------|---------|
| GB/T 32960.1-2025 | 总则 | 全系列（替代2016版）| 4.3 平台数据要求，5.2 数据传输对象，5.3.4 加密要求 |
| GB/T 32960.2-2025 | 车载终端 | TBOX硬件要求 | 终端功能要求、安全要求 |
| GB/T 32960.3-2025 | 通信协议与数据格式 | 数据交互 | 6.1 报文结构，7.1~7.6 数据单元格式 |
| GB/T 32960.4-2025 | 合格性试验 | 型式认证 | 认证测试规程 |

### 1.1 GB/T 32960.3-2025 与旧版主要变化（2026年12月1日实施）

| 变化点 | 旧版（2016）| 新版（2025）| 影响 |
|--------|-----------|------------|------|
| 数据传输加密 | 推荐使用 | **必须使用**（SM2/SM3/SM4）| 安全设计必须实现 |
| 企业平台数据要求 | 无 | **新增4.3节** | 平台接口需重新定义 |
| 公共平台接入要求 | 无 | **新增4.4节** | 需满足公共平台协议 |
| 动力电池单体电压 | 必选 | **已删除** | 仅保留总电压 |
| 登入登出超时 | 未规定 | **明确规定超时时间** | 实现需严格对齐 |

### 1.2 合规检查要点

```
GB/T 32960 合规检查清单：

□ 车载终端必须支持 GB/T 32960.3 规定的数据格式
□ 登入：0x01，登出：0x03，实时数据：0x02，报警数据：0x04
□ 数据加密使用国密算法（SM2/SM3/SM4）
□ 本地存储容量 ≥ 72小时数据（按1条/秒计）
□ 定位数据上报周期 ≤ 1秒
□ 报警触发后 30秒内上报
□ 终端必须具备防拆报警功能
□ 数据补传：断网恢复后自动补传缓存数据
□ SN/ICCID 在线注册功能
□ 终端版本号可远程查询
□ 具备远程重启功能（平台触发）
□ 具备远程参数配置功能
□ 具备身份鉴别（双向认证）
□ SIM 卡实名认证支持
```

## 2. TAF 车载TBOX信息安全技术要求（进网许可前提）

> 来源：电信终端产业协会标准 T/TAF XXX-XXXX（2024版）
> **进网许可必须通过此标准的安全评估**

### 2.1 安全要求章节映射

| 测试编号 | 测试项目 | 对应设计要求 |
|---------|---------|------------|
| 6.1.1 | 防拆报警/数据自毁 | 物理安全 → 壳体防拆开关触发报警 |
| 6.1.2 | 调试接口禁用 | 调试模式 → 上市前必须关闭JTAG/串口 |
| 6.1.3 | 密钥安全管理 | 密钥 → 分发/存储/销毁全程安全 |
| 6.1.4 | 防侧信道攻击 | 关键算法 → SE芯片保护根密钥 |
| 6.2.1 | 可执行文件类型限制 | 软件安全 → 白名单机制 |
| 6.2.6 | CAN总线隔离 | 总线安全 → 内网CAN与外部接口隔离 |
| 6.3.1 | 签名认证机制 | 应用签名 → 未经签名不得执行 |
| 6.5.1 | 双向身份认证 | 通信安全 → 平台与终端双向鉴权 |
| 6.5.2 | CAN总线防DoS | 总线保护 → 控制策略避免拥塞 |
| 6.5.3 | CAN无透传 | 总线功能 → 仅业务相关功能，禁止透传 |

### 2.2 PRD 中必须体现的信息安全章节

```markdown
## X 信息安全要求

### X.1 身份鉴别
- 平台对终端：基于SM2证书的双向身份认证
- 终端对平台：登录时报文签名
- 密钥更新周期：≤12个月或10000次会话

### X.2 数据安全
- 传输加密：SM4国密算法，密钥长度≥128bit
- 存储加密：敏感数据（位置/身份）本地加密存储
- 完整性：每条报文含SM3消息认证码

### X.3 软件安全
- 签名认证：所有可执行文件必须经OEM签名
- 升级包：必须携带版本号+数字签名
- 回滚保护：升级失败自动回滚，不影响车辆运行

### X.4 总线安全
- CAN隔离：内部CAN与外部通信接口电气隔离
- 防注入：仅接收白名单CAN ID，不响应非法报文
- 防DoS：发送频率限制（≤100帧/秒）
```

## 4. 功能安全（ISO 26262 / GB/T 34590）

### 4.1 ASIL 等级建议

| TBOX 功能 | 建议 ASIL | 理由 |
|---------|----------|------|
| 车辆数据采集（标准协议）| ASIL A | 实时监控，不控制车辆 |
| 远程控车（解锁/启动）| ASIL B | 影响车辆安全操作 |
| 碰撞报警触发 | ASIL B | 直接影响救援 |
| OTA 升级 | QM（质量管理）| 不影响安全功能 |
| GPS 定位 | QM | 信息显示，无控制功能 |

### 4.2 安全设计要求

```markdown
## X 功能安全要求

### X.1 故障检测
- CAN 总线故障检测：超时/位翻转/CRC错误
- 通信链路故障：TCP心跳超时（≥60s）→ 触发重连
- 存储故障：数据写入校验和，读取失败告警
- 看门狗：硬件看门狗（独立于MPU）+ 软件任务监控

### X.2 故障响应
- CAN故障 → 进入安全模式（停止发送，保持监听）
- 通信故障 → 本地缓存数据，断网后自动补传
- 存储故障 → 切换到备用存储区域
- 看门狗复位 → 记录复位原因到日志，重连网络

### X.3 降级策略
- 4G断连 → 保持CAN数据本地存储
- GPS失锁 → 惯性导航航位推算（≤60秒）
- 电池电压低 → 进入深度休眠，减少发送频率
```

## 5. AUTOSAR 开发规范（软件架构参考）

### 5.1 软件架构分层

```
┌─────────────────────────────────────────┐
│           Application Layer              │  ← SWC（软件组件）
├─────────────────────────────────────────┤
│        Runtime Environment (RTE)         │  ← AUTOSAR接口
├─────────────────────────────────────────┤
│     Basic Software (BSW)                 │
│  ┌──────────┬───────────┬─────────────┐ │
│  │   I/O   │  COMM     │   MEM       │ │  ← ECU Abstraction
│  ├──────────┼───────────┼─────────────┤ │
│  │  MCAL   │  CAN/LIN  │  Flash Drv  │ │  ← Microcontroller Abstraction
│  └──────────┴───────────┴─────────────┘ │
├─────────────────────────────────────────┤
│           Microcontroller                │
└─────────────────────────────────────────┘
```

### 5.2 SRS 中必须描述的接口

| 接口类型 | 必须描述的内容 |
|---------|-------------|
| CAN Interface | CAN ID列表、帧类型（标准/扩展）、波特率、TX/RX滤波器 |
| NVRAM Interface | 数据项ID、写入策略（同步/异步）、ECC配置 |
| COM Interface | 信号分组、周期、Deadline监控配置 |
| Vehicle Interface | 网络管理（Nm）状态机、功耗状态 |
| SecOC Interface | 消息认证、Freshness Value管理 |
